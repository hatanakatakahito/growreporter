# vivid Phase 2 A/B テスト評価ルーブリック

最終更新: 2026-05-06
対象: vivid Phase 2（improvementKnowledge RAG 注入）の本番デプロイ判定

## テスト目的

RAG 注入 ON / OFF で AI 改善提案の品質に **有意な向上** があり、かつ **バイアス（業界平均化・天井設定・責任転嫁・目標化）が発生していない** ことを検証する。

## テスト方法

### 1. テストサイト選定（5件）
業種多様性を確保するため、以下の業種カバレッジを優先:

| 業種 | improvementKnowledge データ件数の目安 |
|---|---|
| 不動産・建設 | 多（Step 1 でヒット見込み） |
| メーカー | 多 |
| IT・通信 | 中（Step 1 で 3-5件） |
| 医療・介護 | 中 |
| コンサル・士業 | 少（Step 2-3 にフォールバック想定） |

**選定基準**:
- 業種 4軸（businessModel / industryMajor / industryMinor / siteRole）が設定済
- 直近30日にデータがある（GA4 セッション > 100）
- 既存改善案件が 5件以上ある（Jaccard 重複排除との相互作用検証用）
- Grow Group 社内テスト用サイトを優先（顧客本番サイトは避ける）

### 2. 各サイトで2セットの改善提案を生成

#### Set A: 注入 OFF
1. featureFlags の `improvementKnowledgeRagInjection.enabled` を `false` に設定
2. キャッシュクリア（`clearAllAICache` callable 呼び出し or Firestore Console から削除）
3. 改善提案生成（10件程度）
4. 出力を保存

#### Set B: 注入 ON
1. featureFlags を `true` に設定
2. キャッシュクリア
3. 改善提案生成（10件程度）
4. 出力を保存

→ 5サイト × 2セット = **10セット（合計 100件程度の改善提案）**

## 評価ルーブリック

### 評価カテゴリ A: 品質向上（ON > OFF を期待）

#### A1. 具体性（Specificity）
- 改善提案に **本サイト固有の URL・文言・要素名** が含まれている
- 「ファーストビュー」のような抽象表現でなく「/services/page1 の H1 を『〇〇』に変更」のような具体的指示

| スコア | 基準 |
|---|---|
| 5 | 全提案で具体的な URL・文言が引用されている |
| 4 | 80%以上で具体性あり |
| 3 | 50-80%で具体性あり |
| 2 | 30-50%で具体性あり |
| 1 | 30%未満、抽象的提案ばかり |

#### A2. 優先度の根拠（Priority Rationale）
- 各提案の `priority` (high/medium/low) に納得性のある根拠が示されている
- 「業界での実績」「サイト固有の数値」を踏まえて優先度が決まっている

| スコア | 基準 |
|---|---|
| 5 | すべての提案に明確な優先度根拠が示されている |
| 4 | 80%以上で根拠あり |
| 3 | 50-80% |
| 2 | 30-50% |
| 1 | 30%未満、優先度が機械的 |

#### A3. 目標水準の高さ（Aspiration Level）
- 改善後の期待効果が **同業他社のトップ層水準** や、**自社サイトの上限可能性** を示唆している
- 「業界平均並みに」ではなく「業界の上位水準を目指して」のニュアンス

| スコア | 基準 |
|---|---|
| 5 | 多くの提案で野心的な改善目標を提示 |
| 4 | 80%以上で野心的 |
| 3 | 50-80% |
| 2 | 30-50% |
| 1 | 平均値・現状維持的な目標が大半 |

#### A4. 業界文脈の活用（Industry Context Usage）
- 「同業他社で〜」「業界の傾向として〜」のような表現が **適切に** 入っている
- 業界文脈が「補助情報」として扱われ、本サイトデータが主役

| スコア | 基準 |
|---|---|
| 5 | 業界文脈表現が自然に入り、本サイト議論が主役 |
| 4 | 業界文脈ありだが時々強引 |
| 3 | 業界文脈あるが少ない or 浮いている |
| 2 | 業界文脈ほぼなし |
| 1 | なし |

→ ON のみで期待される項目（OFF では当然 1 になる）

---

### 評価カテゴリ B: バイアス検出（ON で 0件であること必須）

#### B1. 業界平均の目標化（FAIL）
- ❌「業界平均を目指しましょう」「業界中央値レベルにする」
- ❌「業界平均と同等なので問題ありません」「業界水準を達成できています」

→ **ON で 1件でも検出 → 即刻ロールバック検討**

#### B2. 生の統計値出力（FAIL）
- ❌「業界平均は 53%」「N=10 サンプル」「+35% 改善」
- ❌「達成度 exceeded」「overallScore 4.5」
- ❌「弊社調べ」「Grow Group 集計」

→ **ON で 1件でも検出 → プロンプト調整必須**

#### B3. 責任転嫁（FAIL）
- ❌「あなたのサイトも業界全体と同じ問題を抱えている」
- ❌「業界全体が低水準なので、業界平均でも問題ありません」

→ **ON で 1件でも検出 → プロンプト調整必須**

#### B4. 業界横断的提案の強要（FAIL）
- ❌ 本サイトに明らかに不適合な施策を「同業がやっているから」だけで提案
- 例: B2B サイトに B2C 向けの「LINE 友達登録キャンペーン」を強引に提案

→ **ON で 1件でも検出 → プロンプト調整推奨**

#### B5. ネガティブトーン過多（WARNING）
- ⚠️「あなたのサイトは業界下位です」「全体的に劣っています」のような落胆を誘う表現
- 業界文脈との対比が意欲を削ぐ方向に

→ **ON で 3件以上検出 → プロンプト調整推奨**

## 合格判定基準

### 必須条件（すべて満たすこと）
- [ ] B1（業界平均目標化）: 全 50件で 0 件
- [ ] B2（生の統計値出力）: 全 50件で 0 件
- [ ] B3（責任転嫁）: 全 50件で 0 件
- [ ] B4（業界横断的強要）: 全 50件で ≤ 2件
- [ ] B5（ネガティブトーン過多）: 全 50件で ≤ 5件

### 望ましい条件（過半数で達成）
- [ ] A1（具体性）: ON の平均 ≥ 4 かつ OFF より 1ポイント以上向上
- [ ] A2（優先度根拠）: ON の平均 ≥ 4 かつ OFF より 1ポイント以上向上
- [ ] A3（目標水準）: ON の平均 ≥ 3.5 かつ OFF より 0.5ポイント以上向上
- [ ] A4（業界文脈活用）: ON の平均 ≥ 3.5

### 総合判定

| 結果 | 判定 |
|---|---|
| 必須条件すべて満たす + 望ましい条件 4/4 達成 | ✅ 本番デプロイGO |
| 必須条件すべて満たす + 望ましい条件 2-3/4 | ⚠️ プロンプト微調整後に再テスト |
| 必須条件で 1件でも違反 | ❌ プロンプト修正必須、再テスト |

## レビュー担当

A/B テストの評価は **2名以上の人手レビュー** を推奨（1名だと主観バイアスが入る）。

評価担当の例:
- Grow Group のコンサル担当者
- AI 出力品質を見慣れている開発者

レビューの最終判定は **両名一致** が望ましい。意見が分かれた項目は議論してコンセンサスを形成。

## トラブルシューティング

### Q: improvementKnowledge データ不足で Set B が空になる

A: テストサイトの業種を、improvementKnowledge データが多い業種（不動産・メーカー）に寄せる。または improvementKnowledge にテストデータを seed する（[seedTestImprovementKnowledge.mjs](../functions/src/scripts/seedTestImprovementKnowledge.mjs) 参照）

### Q: Set A と Set B で全く同じ出力が返る

A: キャッシュが効いている可能性。`clearAllAICache` を実行してから再生成。

### Q: B1-B5 のバイアスが検出された

A: プロンプトの該当ルールを強化する。例えば B1 検出時は templates.js の「利用ルール」に「業界平均を目標として書かない」をより強い表現に書き換える。
